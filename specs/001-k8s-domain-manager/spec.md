# 功能規格:Kubernetes 域名管理器

**功能分支**: `001-k8s-domain-manager`
**建立日期**: 2025-11-07
**狀態**: Draft
**輸入**: 使用者描述:"我想要做一個域名管理器,然後我想部署在Vultr 或類似平台的K8s上,可以把外部的域名導引進來,然後在內部分派給不同的服務使用,而且可以做到域名或子域名SSL認證,可能可以透過Let's Encrypt的自動認證,或是自行提供的憑證。然後也影簡單的DNS,可以管理子域名等等的。然後這專案可以包成Helm,方便在Lens上面管理。然後要假設使用者幾乎沒有K8s知識,所以不要讓設定太難,最好有開箱即用的感覺。"

## 釐清事項

### 會議 2025-11-07

- Q: 管理介面的存取控制機制? → A: 簡單的使用者名稱/密碼驗證(安裝時設定初始密碼)
- Q: 系統通知機制(憑證續約失敗、速率限制等)? → A: 僅在 Web 管理介面中顯示警告和通知
- Q: 資料持久化方式(域名配置、SSL 憑證等)? → A: 使用內嵌的 SQLite 資料庫配合 PersistentVolume
- Q: DNS 記錄的作用範圍和解析方式? → A: 不提供 DNS 記錄管理功能,使用者在域名註冊商自行管理 DNS
- Q: 服務發現與選擇機制? → A: 系統自動發現並列出所有可用的服務,可透過標籤(label)篩選
- Q: 是否提供程式化介面(API/MCP)供外部操作? → A: 提供 RESTful API 和 MCP (Model Context Protocol) 伺服器供外部工具和 AI 助理操作

## 使用者情境與測試 *(必要)*

### 使用者故事 1 - 透過 Helm 快速部署域名管理器 (優先級: P1)

系統管理員想要在 Kubernetes 叢集上快速部署域名管理器,無需深入了解 Kubernetes 的複雜設定。使用者可以透過 Lens 或 Helm CLI 在幾分鐘內完成安裝,並開始管理域名。

**為何是此優先級**: 這是最基本的功能,沒有部署就無法使用任何其他功能。簡化部署流程是吸引非 Kubernetes 專家使用的關鍵。

**獨立測試**: 可以透過在全新的 Kubernetes 叢集上執行 `helm install` 命令,然後存取管理介面來完整測試此功能,並驗證服務已正確啟動。

**驗收情境**:

1. **Given** 使用者有一個運行中的 Kubernetes 叢集, **When** 使用者執行 `helm install domain-manager ./chart` 命令, **Then** 系統在 5 分鐘內完成安裝並顯示存取管理介面的指引
2. **Given** 使用者在 Lens 介面中, **When** 使用者透過 Helm Charts 目錄安裝域名管理器, **Then** 安裝過程顯示清晰的進度和狀態
3. **Given** 域名管理器已安裝, **When** 使用者存取管理介面, **Then** 看到歡迎畫面和快速入門指南

---

### 使用者故事 2 - 設定外部域名並自動取得 SSL 憑證 (優先級: P2)

使用者已經擁有一個外部域名(例如 example.com),想要將此域名導向 Kubernetes 叢集中的服務,並且自動取得 Let's Encrypt SSL 憑證以啟用 HTTPS。

**為何是此優先級**: 這是核心價值功能,讓使用者可以快速將域名與服務連接並確保安全連線。自動化 SSL 憑證管理可以大幅降低技術門檻。

**獨立測試**: 可以透過新增一個域名配置、指向一個測試服務,然後從外部瀏覽器存取 https://example.com 並驗證憑證有效性來測試。

**驗收情境**:

1. **Given** 使用者已在域名註冊商設定 DNS 指向叢集, **When** 使用者在管理介面輸入域名 example.com 並從自動發現的服務列表中選擇目標服務, **Then** 系統自動配置路由並在 10 分鐘內取得 SSL 憑證
2. **Given** SSL 憑證即將到期(30 天內), **When** 系統檢查憑證狀態, **Then** 自動續約憑證而無需使用者介入
3. **Given** 使用者偏好使用自己的憑證, **When** 使用者上傳自訂的憑證和私鑰, **Then** 系統使用該憑證並停止自動憑證管理

---

### 使用者故事 3 - 管理子域名並分派到不同服務 (優先級: P3)

使用者想要為不同的應用程式建立子域名(例如 api.example.com、admin.example.com),並將每個子域名分派到叢集中的不同服務。

**為何是此優先級**: 子域名管理擴展了系統的靈活性,讓使用者可以為多個服務提供獨立的進入點。這是在基本域名管理之後的自然擴展。

**獨立測試**: 可以透過新增多個子域名配置,每個指向不同的服務,然後分別存取各個子域名來驗證路由正確性。

**驗收情境**:

1. **Given** 主域名 example.com 已配置, **When** 使用者新增子域名 api.example.com 並從服務列表中選擇目標服務(可使用標籤篩選), **Then** 子域名在 5 分鐘內可用並自動取得獨立的 SSL 憑證
2. **Given** 使用者有多個子域名, **When** 使用者在管理介面查看域名列表, **Then** 清楚顯示每個域名/子域名及其對應的服務和憑證狀態
3. **Given** 使用者想要移除子域名, **When** 使用者刪除子域名配置, **Then** 系統自動清理相關路由和憑證

---

### 使用者故事 4 - 透過 API 或 MCP 程式化操作 (優先級: P4)

開發者或 AI 助理想要透過程式化方式(API 或 MCP)管理域名配置,以便整合到自動化工作流程或透過 Claude Desktop 等工具操作。

**為何是此優先級**: 這是進階功能,讓系統可以整合到更廣泛的工具鏈中。對於基本使用不是必需的,但對於自動化和 AI 輔助操作很有價值。

**獨立測試**: 可以透過 curl 或其他 HTTP 客戶端呼叫 API 端點,或透過支援 MCP 的客戶端(如 Claude Desktop)連接並執行操作來測試。

**驗收情境**:

1. **Given** 系統已部署並運行, **When** 開發者使用 API 金鑰透過 REST API 新增域名配置, **Then** 域名成功配置並返回配置詳情
2. **Given** 系統啟用 MCP 伺服器, **When** AI 助理透過 MCP 協議查詢現有域名列表, **Then** 返回所有域名及其狀態的結構化資料
3. **Given** 開發者想要自動化憑證管理, **When** 透過 API 查詢即將到期的憑證, **Then** 返回所有 30 天內到期的憑證列表

---

### 邊界情況

- 當 Let's Encrypt 速率限制達到時,系統在管理介面顯示警告並建議使用自訂憑證
- 當域名的 DNS 設定不正確(未指向叢集)時,系統自動偵測並在管理介面提供診斷資訊
- 當使用者嘗試新增已存在的域名時,系統阻止操作並在介面顯示錯誤訊息
- 當目標服務在 Kubernetes 中不存在或無法存取時,系統在介面標記該域名為錯誤狀態並提供診斷建議
- 當 SSL 憑證續約失敗時,系統在管理介面顯示警告並提供手動重試或上傳自訂憑證的選項
- 當多個域名指向同一個服務時,系統在服務詳情頁面列出所有關聯的域名

## 需求 *(必要)*

### 功能需求

- **FR-001**: 系統必須提供 Helm Chart 以便在 Kubernetes 叢集上一鍵安裝
- **FR-002**: 系統必須提供 Web 管理介面,讓使用者可以管理域名和服務映射,並透過使用者名稱/密碼進行身份驗證(初始密碼在安裝時設定)
- **FR-003**: 系統必須支援外部域名(使用者已擁有)導向叢集內服務,透過自動發現機制列出所有可用服務,並支援透過標籤篩選
- **FR-004**: 系統必須自動透過 Let's Encrypt 取得和續約 SSL 憑證
- **FR-005**: 系統必須允許使用者上傳自訂的 SSL 憑證和私鑰
- **FR-006**: 系統必須支援子域名管理,每個子域名可獨立配置
- **FR-007**: 系統必須顯示每個域名的 SSL 憑證狀態(有效期、到期日)
- **FR-008**: 系統必須自動偵測 DNS 配置問題並提供診斷資訊
- **FR-009**: 系統必須提供域名到服務的映射關係清單檢視
- **FR-010**: 系統必須在安裝後提供快速入門指南和範例配置
- **FR-011**: 系統必須支援在 Lens 中進行管理和監控
- **FR-012**: 系統必須驗證域名所有權後才允許取得 SSL 憑證
- **FR-013**: 系統必須處理憑證續約失敗並在 Web 管理介面中顯示警告通知
- **FR-014**: 系統必須允許刪除域名配置並自動清理相關資源
- **FR-015**: 系統必須提供簡易的備份功能,允許使用者下載 SQLite 資料庫檔案作為備份
- **FR-016**: 系統必須提供 RESTful API,支援域名配置的 CRUD 操作,並使用 API 金鑰進行身份驗證
- **FR-017**: 系統必須提供 MCP (Model Context Protocol) 伺服器,讓 AI 助理可以查詢和管理域名配置

### 關鍵實體

- **域名配置 (Domain Configuration)**: 代表一個域名或子域名的完整配置,包含域名名稱、目標服務、SSL 憑證類型(自動/手動)、憑證狀態、建立和更新時間。儲存在 SQLite 資料庫中。
- **SSL 憑證 (SSL Certificate)**: 代表一個 SSL/TLS 憑證,包含憑證來源(Let's Encrypt/使用者上傳)、有效期、到期日、狀態(有效/即將到期/已過期)。憑證內容儲存在 SQLite 資料庫中。
- **服務映射 (Service Mapping)**: 代表 Kubernetes 叢集內的服務,包含服務名稱、命名空間、連接埠、標籤、健康狀態。從 Kubernetes API 動態查詢,不持久化儲存。系統自動發現所有服務並支援透過標籤篩選。
- **診斷記錄 (Diagnostic Log)**: 記錄域名配置過程中的問題和警告,包含時間戳、嚴重程度、訊息內容。儲存在 SQLite 資料庫中。
- **管理員帳戶 (Admin Account)**: 儲存管理員的使用者名稱和加密後的密碼。儲存在 SQLite 資料庫中。
- **API 金鑰 (API Key)**: 用於 RESTful API 和 MCP 身份驗證的金鑰,包含金鑰值、建立時間、最後使用時間。儲存在 SQLite 資料庫中。

## 成功標準 *(必要)*

### 可衡量成果

- **SC-001**: 不熟悉 Kubernetes 的使用者可以在 10 分鐘內完成系統安裝並新增第一個域名
- **SC-002**: 系統在 10 分鐘內自動完成 SSL 憑證申請和配置
- **SC-003**: 憑證自動續約成功率達到 99% 以上
- **SC-004**: 使用者新增域名後,在外部可以在 15 分鐘內透過 HTTPS 存取對應服務
- **SC-005**: 系統支援至少 100 個不同域名和子域名的管理
- **SC-006**: 90% 的使用者可以在不查閱詳細文件的情況下完成基本域名設定
- **SC-007**: 系統能夠在 DNS 配置錯誤時,在 5 分鐘內提供診斷資訊
- **SC-008**: 介面回應時間(頁面載入、操作反饋)保持在 2 秒內

## 假設

1. 使用者已有運行中的 Kubernetes 叢集(版本 1.20 或更高)
2. 叢集具有對外的 LoadBalancer 或 NodePort 存取能力
3. 使用者已在域名註冊商處擁有域名,並有能力修改 DNS 設定
4. 叢集有足夠的資源運行域名管理器(至少 1 CPU 核心和 512MB 記憶體)
5. 叢集可以存取網際網路以與 Let's Encrypt 服務通訊
6. 使用者使用的 Kubernetes 環境支援標準的 Ingress 控制器
7. 叢集支援 PersistentVolume 以提供資料持久化儲存

## 範圍限制

### 包含在範圍內

- Helm Chart 形式的部署方案
- Web 管理介面
- RESTful API(支援 CRUD 操作)
- MCP (Model Context Protocol) 伺服器
- Let's Encrypt 自動憑證管理
- 使用者自訂憑證支援
- 域名到服務的路由配置
- 子域名管理
- 域名配置的診斷和健康檢查
- 快速入門文件和範例
- 資料庫備份功能

### 不包含在範圍內

- DNS 記錄管理功能(使用者需自行在域名註冊商管理 DNS)
- DNS 伺服器功能
- 域名註冊商 API 整合
- 流量分析和監控儀表板
- 多叢集管理
- 多使用者權限管理(僅支援單一管理員帳戶)
- 進階的 Ingress 配置(如流量分割、藍綠部署)
- 自動化的域名註冊
- 內建的防火牆或 DDoS 防護

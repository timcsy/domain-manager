# 原則

## 根公理

**Kubernetes 資源是唯一的事實來源，所有操作方式只是它的不同投影。**

不論操作從哪個介面發起，最終狀態必須回歸到 K8s 資源。
任何與 K8s 資源不一致的狀態都是暫時的、待同步的。

## 衍生原則

### 1. 宣告式優先

- **推導自**：根公理
- **意義**：任何操作最終必須轉化為 Kubernetes 資源的變更，而非繞過它
- **如何應用**：UI、API 的每個寫入操作都必須產生或修改對應的 K8s 資源；不存在「只在 UI 生效」的狀態

### 2. 多入口對等

- **推導自**：根公理
- **意義**：Web UI、CLI、API、直接 kubectl 都是同等地位的設定方式，沒有特權入口
- **如何應用**：從任何入口做的變更，在其他入口都必須可見；系統需要協調機制處理並行修改，衝突時以 K8s 資源為準

### 3. 預設先行，細節按需

- **推導自**：根公理 + 使用者體驗不妥協
- **意義**：能用預設值的地方就用預設值，讓 DevOps 最少操作就能完成任務。底層的 K8s 細節不主動暴露，但使用者要求時必須完整呈現
- **如何應用**：新增 domain 時自動帶入合理預設（憑證類型、ingress class 等）；UI 預設顯示簡潔視圖，提供「展開詳情」查看完整 K8s 資源
